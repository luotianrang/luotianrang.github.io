独立完成 StudyApp：一个从需求重构到多用户隔离的学习助手项目复盘

最近我独立完成了一个在线学习工具项目 StudyApp。这个项目基于 FastAPI + SQLite + 原生 HTML/CSS/JS 开发，核心围绕“预设书库、我的学习、学习计划、后台管理”展开。整个开发过程中，我在 Codex 和 DeepSeek V4 Pro 的辅助下完成需求拆解、功能实现、结构重构与 bug 排查，但项目设计、取舍和落地都由我个人独立推进。

这篇文章既是一次项目复盘，也是一份比较完整的阶段性总结。我想重点记录三件事：

  1. 这个项目为什么这样设计
  2. 我具体完成了哪些核心模块
  3. 它体现了我哪些工程能力和问题解决能力

一、项目背景：我想做的不是“演示页”，而是一个能跑通流程的小产品

我一开始并不想只做一个静态前端页面，或者写几个独立接口做技术展示，而是想做一个真正有用户路径的学习工具。

最终我把 StudyApp 的业务流程设计成下面这样：

  1. 用户注册 / 登录
  2. 浏览预设书库
  3. 把书加入“我的学习”
  4. 生成学习计划
  5. 按天学习并标记完成

从这个流程可以看出，我想做的是一个有“内容、用户、计划、进度”完整闭环的小型产品，而不是单点功能 demo。

二、技术栈：选轻量方案，但把关键链路做完整

这个项目我采用的是一套比较轻量但完整的技术栈：

  • 后端：Python + FastAPI
  • 数据库：SQLite
  • 前端：原生 HTML / CSS / JavaScript SPA
  • 认证：JWT（python-jose + bcrypt）
  • 数据导入：DOCX 解析

我没有一开始就上更重的框架，而是优先保证自己能真正理解每一层是怎么工作的。对我来说，这种方式更适合练习接口设计、数据库建模、前后端联调和实际排错。

三、最关键的一次产品决策：从“上传分析”重构为“预设书库”

这个项目里，我觉得最有价值的一次决策不是写了哪个功能，而是主动推翻了原来的方案。

最早的思路是让用户上传 PDF，系统再去分析内容并生成学习数据。但我很快发现两个问题：

  • 分析速度慢
  • 结果质量不稳定

如果一个学习工具连最开始的导入都做不好，后面再多功能也很难成立。所以我决定做一次明确的产品重构，把模式改成“预设书库”。这次调整包括：

  • 删除上传书籍功能和相关页面
  • 新增“我的学习”页面，并设为默认首页
  • 重写书库页面，改为纯预设书展示
  • 新增 UserBook 模型和 API，支持加入学习 / 移除学习 / 列表查询
  • 保留知识点查询能力，把重点转回学习流程本身

这次重构让我很直观地意识到，做项目时不能只想着“把功能堆上去”，而是要围绕“是否真的可用”做取舍。这其实也是工程思维里非常重要的一部分。

四、我独立完成的核心模块

1. 用户注册、登录与鉴权流程

为了让项目不再是“单人演示模式”，我实现了完整的用户认证流程：

  • 注册接口
  • 登录接口
  • 获取当前用户信息接口
  • 前端登录 / 注册页面
  • JWT Token 鉴权

这部分虽然属于基础能力，但它决定了整个系统是否具备真正的多用户使用前提。

2. 学习计划与用户数据隔离

我在这个项目里比较重视的一次后端改造，是把原本写死的用户逻辑彻底改成了按用户隔离的数据流。

之前某些功能默认使用固定的 user_id = 1，这样会导致不同用户之间的数据边界不清晰。后来我完成了这几项改造：

  • 为 StudyPlan 和 NotificationSetting 添加 user_id 字段
  • 让 plan.py 所有端点统一接入 get_current_user()
  • 创建计划时写入当前用户 ID，列表查询时按用户过滤
  • 通知设置按用户隔离
  • 在数据库初始化中加入自动迁移逻辑，避免改表时丢数据

这一部分的意义不只是“功能能用了”，而是我第一次比较完整地处理了多用户系统里的权限边界和数据归属问题。

3. 后台管理：书籍、章节、知识点增删改查

为了让预设书库模式真正可持续，我还做了网页端后台管理，支持：

  • 书籍管理
  • 章节管理
  • 知识点管理

这意味着项目不再只是固定展示几条静态数据，而是具备继续维护和扩展内容的能力。从结构上看，这个项目已经更像一个可以持续迭代的小系统。

4. DOCX 知识图谱导入

项目里我还实现了 DOCX 内容导入，用于批量整理书籍章节与知识点数据。

目前已经导入《高等数学（下册）》知识图谱，共 5 章、50 个知识点。这一步让我把项目推进到了更真实的数据处理场景，而不是停留在手写几条 mock 数据的层面。

五、开发过程中最有代表性的几个问题

我觉得这个项目最有价值的地方之一，是我不是只在“加功能”，而是在不断解决真实开发中会遇到的具体问题。

1. api.js 中多个接口地址写错

我遇到过接口 404、参数传不进去的问题。排查之后发现原因并不复杂，但很隐蔽：多个 URL 被写成了正则字面量，而不是模板字符串。

这类问题让我意识到，前后端联调时很多 bug 不是大逻辑错了，而是很基础但足以让系统失效的细节问题。

2. Python 引号转义与相对导入错误

我也处理过比较典型的后端报错，比如：

  • SyntaxError: unexpected character after line continuation character
  • ImportError: attempted relative import

前者来自错误的引号转义，后者来自相对导入写法不统一。它们都不是业务问题，但会直接影响系统是否能启动，这也逼着我去理解代码结构本身。

3. start.bat 乱码和 LF/CRLF 兼容问题

这个问题对我来说很真实，因为它不是算法问题，而是运行环境兼容问题。

我遇到过：

  • 批处理文件双击后中文乱码
  • 因为 LF 行尾导致命令被错误拆分执行

最后我通过无 BOM 写入和强制使用 \r\n 行尾解决了它。这个过程让我更理解 Windows 环境下脚本执行的细节。

4. 前端结构错误导致整页滚动失效

还有一次前端问题来自一个缺失的闭合 div。看起来只是标签结构小错误，但它直接导致主内容区嵌套异常，最后整个页面的 flex 布局和滚动行为都失效了。

这类问题让我更深刻地体会到，前端页面不是“能显示就行”，结构层面的错误同样会造成系统性影响。

六、这个项目体现了我哪些能力

如果把这个项目放在求职语境里，我觉得它比较能体现我这几方面的能力：

  • 能够独立完成从需求拆解到前后端落地的完整流程
  • 具备基础的后端接口设计、数据库建模和用户鉴权能力
  • 能够理解并处理多用户系统中的数据隔离问题
  • 遇到问题时能定位、排查并修复真实 bug，而不是停留在表面功能展示
  • 能从“功能很多”转向“产品是否可用”的角度思考项目重构

对我来说，StudyApp 的价值不只是“做出了一个项目”，而是它完整展示了我从想法到交付的执行能力。

七、下一步计划

从当前项目状态看，我接下来还准备继续推进这些能力：

  • 重构学习计划生成逻辑，支持新学与复习任务混排
  • 增加 AI 出题、答题、批改等课后练习能力
  • 实现错题本页面
  • 完善用户偏好设置
  • 继续把本地 MVP 闭环打磨完整

这些功能如果继续补齐，StudyApp 会更接近一个真正能长期使用和持续扩展的学习平台。

结语

回头看这个项目，我最满意的并不是某一个页面做得多漂亮，而是我真的把一个想法推进成了一个有完整结构的小型系统。

它让我经历了产品重构、后端改造、登录鉴权、数据隔离、内容管理和持续排错，也让我更明确了自己之后在安全与工程方向继续深入的路径。

如果之后我再做类似项目，我会继续保留这种方式：不只追求“做出来”，而是尽量让每一个项目都更接近真实使用场景。
