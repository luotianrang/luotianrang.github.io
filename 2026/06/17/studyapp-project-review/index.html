从零独立完成一个学习助手项目：StudyApp 开发复盘

最近我完成了一个自己的在线学习工具项目 StudyApp。这个项目并不是简单做一个前端页面或者单独写几个接口，而是从产品思路、数据结构、前后端联动，到用户登录、学习计划、后台管理、知识点导入和 bug 修复，完整走了一遍真正的项目开发流程。

这篇文章想系统记录一下这个项目是怎么做出来的，我做了什么决定，踩了哪些坑，又是怎么一点点把项目从“能跑”推进到“更像一个完整产品”的。

项目定位

StudyApp 是一个“预设书库 + 我的学习”的在线学习工具。

它的核心思路不是让用户上传一本书之后临时分析，而是由管理员提前整理好书籍、章节和知识点，用户进入系统后可以直接浏览书库，把感兴趣的内容加入自己的学习列表，再根据内容生成学习计划并按天学习。

项目目前的完整业务流是：

  1. 用户注册或登录
  2. 浏览预设书库
  3. 加入“我的学习”
  4. 生成学习计划
  5. 每天学习并标记完成

从产品角度看，这个项目更接近一个真正能继续扩展的学习平台，而不是一次性的功能 demo。

技术栈

这个项目采用的是比较轻量但完整的一套技术栈：

  • 后端：Python + FastAPI
  • 数据库：SQLite
  • 前端：原生 HTML / CSS / JavaScript 单页应用
  • 认证：JWT（python-jose + bcrypt）
  • 数据导入：DOCX 解析

我刻意没有一上来就套用复杂框架，而是选择用更直接的方式把业务真正跑通。这样做的好处是，我能更清楚地理解每个模块之间是如何协作的，而不是依赖脚手架帮我隐藏细节。

从“上传分析”到“预设书库”

这个项目中我觉得最重要的一次产品决策，是把原来的“上传书籍分析”方案彻底重构成“预设书库”模式。

最初的想法是让用户上传 PDF，然后系统自动分析内容并生成学习数据。但实际推进时我发现两个问题：

  • 分析速度慢
  • 生成质量不稳定

这两个问题直接影响用户体验。如果一个学习工具在最开始的导入阶段就又慢又不稳定，那后面所有功能都会失去意义。

所以我做了比较彻底的调整：

  • 删除上传书籍功能及相关页面
  • 新增“我的学习”页面，并设为默认首页
  • 重写书库页面，改成纯预设书籍展示
  • 新增 UserBook 模型与相关 API，支持用户加入学习和移除学习
  • 保留知识点查询能力，把重点放回学习流程本身

这次重构让我更清楚地理解了一件事：做项目时，不是功能越多越好，而是要围绕“真正可用”来做取舍。

我完成了哪些核心模块

1. 用户注册与登录

为了让项目真正具备多用户能力，我实现了完整的 JWT 登录体系：

  • 注册接口
  • 登录接口
  • 获取当前用户信息接口
  • 前端登录注册页面
  • Token 鉴权流程

这部分看起来是基础功能，但它决定了后续所有学习数据是否能真正属于“不同的人”。

2. 全 API user_id 隔离

这是我在项目里很重视的一次后端改造。

在早期版本里，部分数据逻辑默认使用固定 user_id=1，这意味着不同用户之间的数据其实没有完全隔离。后来我把这部分彻底改掉了：

  • 给 StudyPlan 和 NotificationSetting 增加 user_id 字段
  • 让 plan.py 所有端点都接入 get_current_user
  • 创建计划时写入 user_id，查询计划时按用户过滤
  • 通知设置按 user_id 隔离
  • 数据库加入自动迁移逻辑，避免改表时丢数据

这一部分的意义不只是“功能修复”，而是让我第一次比较完整地处理了一个项目中的数据权限边界问题。

3. 后台管理与知识点维护

为了让预设书库模式真正可持续，我还做了后台管理相关能力，支持：

  • 书籍增删改查
  • 章节增删改查
  • 知识点增删改查

这部分功能的价值在于，项目不再只是展示固定数据，而是具备继续维护内容的能力。换句话说，它更像一个可以被运营和扩展的系统。

4. DOCX 导入知识图谱

项目里我还实现了 DOCX 内容导入，用来批量整理章节与知识点结构。

目前已经导入的数据包括《高等数学（下册）》知识图谱，共 5 章、50 个知识点。这一步让我更接近“真实数据入库”的场景，而不是只在页面里手写几条假数据演示。

开发过程中印象最深的几个坑

这个项目里我不只是在写功能，也花了很多时间处理真实开发里常见但很麻烦的问题。下面几个坑让我印象特别深。

1. api.js 里有 10 个接口地址写错

问题现象是接口 404，或者参数明明传了却接收不到。排查后发现原因很典型：URL 写成了正则字面量，而不是模板字符串。

这个问题涉及 addToMyLearning、getBook、getPlan、deletePlan 等多个方法。修复方式虽然不复杂，但它提醒我，前后端联调时最怕的往往不是大逻辑，而是这些基础但隐蔽的细节错误。

2. Python 引号转义和相对导入错误

我遇到过 `SyntaxError: unexpected character after line continuation character`，原因是文件里写入了 \" 而不是正常的双引号。

另外还有 analysis.py 的相对导入错误，导致出现 ImportError: attempted relative import。这些问题都不是业务层面的 bug，但会直接让项目跑不起来。

3. start.bat 乱码与 LF/CRLF 问题

这部分是我以前不太会注意，但这次真正踩到的 Windows 兼容问题。

我遇到过两个很具体的现象：

  • 批处理双击后中文乱码
  • 因为行尾是 LF 而不是 CRLF，命令被拆碎执行，出现奇怪报错

最后通过无 BOM 写入和强制使用 \r\n 行尾解决了。这种问题虽然不“高级”，但特别考验耐心和细节。

4. 页面结构错误导致整体滚动失效

还有一次比较典型的前端 bug，是 index.html 少了一个闭合的 div，导致主内容区嵌套异常，最后整个页面 flex 布局失效、页面滚动也出了问题。

这个 bug 给我的感受很深，因为它说明前端页面看似“只是少一个标签”，但在布局系统里会放大成整页失效。

这个项目让我真正学到的东西

如果只从“我会了什么技术”去总结，这个项目当然让我进一步熟悉了 FastAPI、SQLite、JWT、原生前端和接口联调。

但更重要的是，我觉得自己在下面几个方面进步明显：

  • 更能从产品可用性出发做功能取舍
  • 更理解多用户系统中的数据隔离逻辑
  • 更能独立排查前后端联动中的真实 bug
  • 更能把一个项目从“功能点堆叠”推进到“结构完整”

这也是为什么我认为 StudyApp 对我来说不是一个简单练手项目，而是一次比较完整的工程实践。

下一步准备继续做什么

从项目文档来看，后续我还准备继续推进这些能力：

  • 重构学习计划生成逻辑，加入新学与复习任务混排
  • 实现课后练习、AI 出题、答题与批改
  • 增加错题本页面
  • 完善用户偏好设置
  • 继续把本地 MVP 闭环打磨完整

这些功能如果继续补齐，StudyApp 会更接近一个真正可以长期使用的学习平台。

结语

回头看这个项目，我最满意的不是“做了多少页面”，而是它让我切实体验到了一个项目从产品调整、后端重构、数据隔离，到持续排错和功能迭代的完整过程。

对我来说，StudyApp 不是“我写了一个页面”，而是“我独立完成了一个能持续演进的全栈项目”。这也是我后面继续学习网络安全和工程实践时，特别重要的一块积累。
